Online Bottleneck Matching

The Problem:

You are given a set S={s1,s2,...,sn} of servers in set locations, and a set R={r1,r2,...,rn} of requests that each need to reach a server. The request locations are random and occur one at a time, and both sets occur in a 2D space. Each server can only service one request, and every request must be serviced (in most cases).

The Goal:

With no knowledge of future request locations, minimize the cost (or distance) of the most expensive server-request match.

Implementation:

Created in Python 3, two versions of the project exist: one small-scale graphical depiction and one larger-scale non-graphical version. The graphical version uses Zelle's graphics.py file to create a plane of 6 randomly placed users. Both versions test several algorithms: greedy-by-distance, optimal, greedy-swapping (in which you can swap two matches as the come in, dependent on a maximum threshold), and the serve-or-skip (in which you can reject a certain number of requests dependent upon a maximum threshold).

All results were compared to the optimal algorithm. Trends continued between input-sets of size 10 to size 750; serve-or-skip resulted in the closest bottleneck to opt, followed by greedy-swapping, and understandably, ending in greedy-by-distance with the largest bottleneck.

Diagonal Dominoes

Diagonal Dominoes is based on a mini-game found in the PS2 video game 'Avatar: the Last Airbender'. Each player begins with five two-toned tiles, each tile's colors a random combination from four pre-set colors.

The human player can start the game, placing any tile in their collection horizontally on the bottom left corner of the board. From there, the players switch off placing tiles horizontally, one row and one column up and to the right of the open tile. Each tile placed must have a color that corresponds to the one on the board, resulting in each column having only one color in it.

At the start of the game, the computer creates a tree of all possible moves. From there, if a path exists that can guarantee the computer's victory, it will take it. If not, the computer will opt for the path that has the most opportunity for human error, as this ups its chances of winning.

The game ends either when a player has no more tiles with the right color to play, or no more tiles at all (this case will only be available to the human player, as they always go first).

The Pond: A Game of Imitation Dynamics and Absorbing Sets

The Pond is a village of players, arranged in a circle around the pond. Each player is either an altruist, giving a set number (1.0) of 'good', or points, to their neighbors at a cost to themselves (0.3), or an egoist, giving nothing away with no cost to themselves.

In the game, there are two rounds: the first is a village of six, where each player affects a neighborhood of one person on either side of themselves. The human user can switch one player per round from an egoist to an altruist, or vice versa. After this is done (or if the user chooses not to switch anyone), the round ends and each player's total, based on the number of altruists in the player's surrounding neighborhood, is tallied.

These tallies are compared to the average total of altruists in each neighborhood and all egoists in each neighborhood. If the average of the other type of player is higher, that player will switch. If not, it will remain what it was.

The second round is larger, with a larger neighborhood, but the goal and gameplay remain the same. The user wins if they can get all players to choose to be altruists. The user loses if all players choose to be egoists.
